# Formato de transacción atómica de Coreth

Se pretende que esta página sea la única fuente de información sobre la forma de serializar transacciones atómicas en `Coreth`. Este documento utiliza el formato de [serialización primitiva](serialization-primitives.md) para el empaquetado y [secp256k1](cryptographic-primitives.md#cryptography-in-the-avalanche-virtual-machine) para la identificación criptográfica del usuario.

## Codec ID

Algunos datos se preparan con un códec ID (unt16) que denota cómo los datos deben ser deserializados. En este momento, la única identificación de códec válido es 0 (`0x00 0x00`).

## Entrada

Las entradas a las transacciones atómicas de Coreth son ya sea un `EVMInput` de esta cadena o un `TransferableInput` (que contiene un `SECP256K1TransferInput`) de otra cadena. El `EVMInput` se usará en `ExportTx` para gastar fondos de esta cadena, mientras que el `TransferableInput` se usará para importar UTXO atómicos desde otra cadena.

### Entrada de la EVM o "EVM Input"

Tipo de entrada que especifica una cuenta de la EVM de la cual deducir los fondos como parte de un `ExportTx`.

#### What EVM Input Contains

Una entrada de la EVM contiene un `address`, `amount`, `assetID` y `nonce`.

* **`Address`** es la dirección en la EVM desde donde se van a transferir los fondos.
* **`Amount`** es el monto del activo que se va a transferir (especificado en nAVAX para AVAX y la denominación más pequeña para todos los otros activos).
* **`AssetID`** es la identificación del activo que se va a transferir.
* **`Nonce`** es el nonce (número que se usa una sola vez) de la cuenta en la EVM que exporta fondos.

#### Especificación Gantt de una Entrada de la EVM

```text
+----------+----------+-------------------------+
| address  : [20]byte |                20 bytes |
+----------+----------+-------------------------+
| amount   : uint64   |                08 bytes |
+----------+----------+-------------------------+
| asset_id : [32]byte |                32 bytes |
+----------+----------+-------------------------+
| nonce    : uint64   |                08 bytes |
+----------+----------+-------------------------+
                      |                68 bytes |
                      +-------------------------+
```

#### Especificación Proto de una Entrada de la EVM

```text
message  {
    bytes address = 1; // 20 bytes
    uint64 amount = 2; // 08 bytes
    bytes assetID = 3; // 32 bytes
    uint64 nonce = 4;  // 08 bytes
}
```

### Entrada Transferible

La entrada transferible cifra un `SECP256K1TransferInput`. Las entradas transferibles o "Transferable inputs" describen un UTXO específico con una entrada de transferencia proporcionada.

#### Que Contiene una Entrada Transferible

Una entrada transferible contiene un `TxID`, `UTXOIndex` `AssetID` y un `Input`.

* **`TxID`** es una matriz de 32 bytes que define desde qué transacción esta entrada está consumiendo una salida.
* **`UTXOIndex`** es un int que define qué UTXO está consumiendo esta entrada en la transacción indicada.
* **`AssetID`** es una matriz de 32 bytes que define a qué activo hace referencia esta entrada.
* **`Input`** es un `SECP256K1TransferInput`, según se define a continuación.

#### Especificación Gantt de una Entrada Transferible

```text
+------------+----------+------------------------+
| tx_id      : [32]byte |               32 bytes |
+------------+----------+------------------------+
| utxo_index : int      |               04 bytes |
+------------+----------+------------------------+
| asset_id   : [32]byte |               32 bytes |
+------------+----------+------------------------+
| input      : Input    |      size(input) bytes |
+------------+----------+------------------------+
                        | 68 + size(input) bytes |
                        +------------------------+
```

#### Especificación Proto de una Entrada Transferible

```text
message TransferableInput {
    bytes tx_id = 1;       // 32 bytes
    uint32 utxo_index = 2; // 04 bytes
    bytes asset_id = 3;    // 32 bytes
    Input input = 4;       // size(input)
}
```

### Entrada de Transferencia SECP256K1

La entrada de una transferencia [secp256k1](https://github.com/ava-labs/avalanche-docs/tree/94d2e4aeddbf91f89b830f9b44b4aa60089ac755/build/cryptographic-primitives/README.md#cryptography-in-the-avalanche-virtual-machine) permite gastar la salida de una transferencia secp256k1 que no se ha gastado.

#### Contenido de la entrada de una transferencia SECP256K1

La entrada de una transferencia secp256k1 contiene `Amount` y `AddressIndices`.

* **`TypeID`** es la identificación para este tipo de entrada. Es `0x00000005`.
* **`Amount`** es un long que especifica la cantidad que esta entrada debería estar consumiendo del UTXO. Debe ser positivo. Debe ser igual a la cantidad especificada en la UTXO.
* **`AddressIndices`** es una lista de ints únicos que definen las claves privadas que se están usando para gastar el UTXO. Cada UTXO tiene un conjunto de direcciones que pueden utilizar la UTXO. Cada int representa el índice de esta matriz de direcciones que firmará esta transacción. La matriz debe ser ordenada de baja a alta.

#### Especificación Gantt de una entrada de transferencia SECP256K1

```text
+-------------------------+-------------------------------------+
| type_id         : int   |                             4 bytes |
+-----------------+-------+-------------------------------------+
| amount          : long  |                             8 bytes |
+-----------------+-------+-------------------------------------+
| address_indices : []int |  4 + 4 * len(address_indices) bytes |
+-----------------+-------+-------------------------------------+
                          | 16 + 4 * len(address_indices) bytes |
                          +-------------------------------------+
```

#### Especificación Proto de una entrada de transferencia SECP256K1

```text
message SECP256K1TransferInput {
    uint32 typeID = 1;                   // 04 bytes
    uint64 amount = 2;                   // 08 bytes
    repeated uint32 address_indices = 3; // 04 bytes + 04 bytes * len(address_indices)
}
```

#### Ejemplo de una Entrada de Transferencia SECP256K1

Hagamos una entrada de pago con:

* **`TypeId`**: 5
* **`Amount`**: 500000000000
* **`AddressIndices`**: [0]

```text
[
    TypeID         <- 0x00000005
    Amount         <- 500000000000 = 0x000000746a528800,
    AddressIndices <- [0x00000000]
]
=
[
    // type id:
    0x00, 0x00, 0x00, 0x05,
    // amount:
    0x00, 0x00, 0x00, 0x74, 0x6a, 0x52, 0x88, 0x00,
    // length:
    0x00, 0x00, 0x00, 0x01,
    // sig[0]
    0x00, 0x00, 0x00, 0x00,
]
```

## Salidas

Las salidas de las transacciones atómicas de Coreth son ya sea un `EVMOutput` que se agrega al saldo de una dirección de esta cadena o un `TransferableOutput` (que contiene un `SECP256K1TransferOutput`) que se trasladará a otra cadena.

La salida de la EVM se usará en `ImportTx` para agregar fondos a esta cadena, mientras que el `TransferableOutput` se utilizará para exportar UTXO atómicos a otra cadena.

### Salida de la EVM

Tipo de salida que especifica un cambio de estado que se le deberá aplicar a una cuenta de la EVM como parte de un `ImportTx`.

#### Que Contiene Una Salida de la EVM

Una salida de la EVM contiene un `address`, `amount` y `assetID`.

* **`Address`** es la dirección en la EVM que recibe los fondos.
* **`Amount`** es el monto del activo que se va a transferir (especificado en nAVAX para AVAX y la denominación más pequeña para todos los otros activos).
* **`AssetID`** es la identificación del activo que se va a transferir.

#### Especificación Gantt de una Salida de la EVM

```text
+----------+----------+-------------------------+
| address  : [20]byte |                20 bytes |
+----------+----------+-------------------------+
| amount   : uin64    |                08 bytes |
+----------+----------+-------------------------+
| asset_id : [32]byte |                32 bytes |
+----------+----------+-------------------------+
                      |                60 bytes |
                      +-------------------------+
```

#### Especificación Proto de una Salida de la EVM

```text
message  {
    bytes address = 1; // 20 bytes
    uint64 amount = 2; // 08 bytes
    bytes assetID = 3; // 32 bytes
}
```

### Salida Transferible

Las salidas transferibles cifran un `SECP256K1TransferOutput` con la identificación de un activo.

#### Que Contiene una Salida Transferible

Una salida transferible contiene un `AssetID` y un `Output` que es un `SECP256K1TransferOutput`.

* **`AssetID`** es una matriz de 32 bytes que define a qué activo hace referencia esta salida.
* **`Output`** es un `SECP256K1TransferOutput` según se define a continuación.

#### Especificación Gantt de una Salida Transferible

```text
+----------+----------+-------------------------+
| asset_id : [32]byte |                32 bytes |
+----------+----------+-------------------------+
| output   : Output   |      size(output) bytes |
+----------+----------+-------------------------+
                      | 32 + size(output) bytes |
                      +-------------------------+
```

#### Especificación Proto de una Salida Transferible

```text
message TransferableOutput {
    bytes asset_id = 1; // 32 bytes
    Output output = 2;  // size(output)
}
```

### Salida de Transferencia SECP256K1

Una salida de transferencia [secp256k1](cryptographic-primitives.md#cryptography-in-the-avalanche-virtual-machine) permite enviar una cantidad de un activo a un conjunto de direcciones después de una hora específica en Unix.

#### Contenido de una salida de transferencia SECP256K1

Una salida de transferencia secp256k1 contiene `TypeID`, `Amount`, `Locktime`, `Threshold` y `Addresses`.

* **`TypeID`** es la identificación para este tipo de salida. Es `0x00000007`.
* **`Amount`** es un long que especifica la cantidad del activo que posee esta salida. Debe ser positivo.
* **`Locktime`** es un long que contiene la marca temporal de Unix después de la cual se puede gastar esta salida. El timestamp unix es específico para el segundo.
* **`Threshold`** es un int que nombra la cantidad de firmas únicas necesarias para gastar la salida. Debe ser menor o igual a la longitud de **`Addresses`**. Si **`Addresses`** está vacío, debe ser 0.
* **`Addresses`** es una lista de direcciones únicas que corresponden a las claves privadas que se pueden utilizar para gastar esta salida. Las direcciones deben estar ordenadas lexicográficamente.

#### Especificación Gantt de una salida de transferencia SECP256K1

```text
+-----------+------------+--------------------------------+
| type_id   : int        |                        4 bytes |
+-----------+------------+--------------------------------+
| amount    : long       |                        8 bytes |
+-----------+------------+--------------------------------+
| locktime  : long       |                        8 bytes |
+-----------+------------+--------------------------------+
| threshold : int        |                        4 bytes |
+-----------+------------+--------------------------------+
| addresses : [][20]byte |  4 + 20 * len(addresses) bytes |
+-----------+------------+--------------------------------+
                         | 28 + 20 * len(addresses) bytes |
                         +--------------------------------+
```

#### Especificación Proto de una salida de transferencia SECP256K1

```text
message SECP256K1TransferOutput {
    uint32 typeID = 1;            // 04 bytes
    uint64 amount = 2;            // 08 bytes
    uint64 locktime = 3;          // 08 bytes
    uint32 threshold = 4;         // 04 bytes
    repeated bytes addresses = 5; // 04 bytes + 20 bytes * len(addresses)
}
```

#### Ejemplo de una Salida de Transferencia SECP256K1

Hagamos una Salida de Transferencia SECP256K1:

* **`Locktime`**: 0

## Transacciones Atómicas

Las Transacciones Atómicas o "Atomic Transactions" se utilizan para mover fondos entre las cadenas. Existen dos tipos `ImportTx` y `ExportTx`.

### ExportTx

Una Transacción Exportar o "ExportTx" es una transacción para exportar fondos de Coreth a una cadena diferente.

#### Que Contiene una Transacción Exportar

Una transacción de exportación (ExportTx) contiene un `typeID`, `networkID`, `blockchainID`, `destinationChain`, `inputs` y `exportedOutputs`.

* **`typeID`** es un int que el tipo para una transacción de exportación. El typeID para una exportTx es 1.
* **`networkID`** es un int que define a qué red de Avalanche se busca emitir esta transacción. Esto podría referirse a la red principal, a la red Fuji, etc. y es diferente a la identificación de la red de la EVM.
* **`blockchainID`** es una matriz de 32 bytes que define a qué blockchain le fue emitida esta transacción.
* **`destinationChain`** es una matriz de 32 bytes que define a qué blockchain exporta fondos esta transacción.
* **`inputs`** es un conjunto de entradas de la EVM para financiar la ExportTx.
* **`exportedOutputs`** es un conjunto de salidas transferibles (TransferableOutputs) que se van a transferir a `destinationChain`.

#### Especificación Gantt de una Transacción Exportar

```text
+---------------------+----------------------+-------------------------------------------------+
| typeID              : int                  |                                        04 bytes |
+---------------------+----------------------+-------------------------------------------------+
| networkID           : int                  |                                        04 bytes |
+---------------------+----------------------+-------------------------------------------------+
| blockchainID        : [32]byte             |                                        32 bytes |
+---------------------+----------------------+-------------------------------------------------+
| destinationChain    : [32]byte             |                                        32 bytes |
+---------------------+----------------------+-------------------------------------------------+
| inputs              : []EvmInput           |                          4 + size(inputs) bytes |
+---------------------+----------------------+-------------------------------------------------+
| exportedOutputs     : []TransferableOutput |                 4 + size(exportedOutputs) bytes |
+----------+----------+----------------------+-------------------------------------------------+
                                             | 80 + size(inputs) + size(exportedOutputs) bytes |
                                             +-------------------------------------------------+
```

### ImportTx

Una Transacción Importar o "ImportTx" es una transacción para importar fondos de Coreth a una cadena diferente.

#### Que Contiene una Transacción Importar

Una transacción de importación (ImportTx) contiene un `typeID`, `networkID`, `blockchainID`, `destinationChain`, `importedInputs` y `Outs`.

* **`typeID`** es un int que el tipo para una transacción de importación. El typeID para un `ImportTx` es 0.
* **`networkID`** es un int que define a qué red de Avalanche se busca emitir esta transacción. Esto podría referirse a la red principal, a la red Fuji, etc. y es diferente a la identificación de la red de la EVM.
* **`blockchainID`** es una matriz de 32 bytes que define a qué blockchain le fue emitida esta transacción.
* **`sourceChain`** es una matriz de 32 bytes que define desde qué blockchain importar fondos.
* **`importedInputs`** es un conjunto de entradas transferibles (TransferableInputs) para financiar la transacción de importación (ImportTx).
* **`Outs`** es un conjunto de salidas de la EVM para importar a esta cadena.

#### Especificación Gantt de una Transacción Importar

```text
+---------------------+----------------------+-------------------------------------------------+
| typeID              : int                  |                                        04 bytes |
+---------------------+----------------------+-------------------------------------------------+
| networkID           : int                  |                                        04 bytes |
+---------------------+----------------------+-------------------------------------------------+
| blockchainID        : [32]byte             |                                        32 bytes |
+---------------------+----------------------+-------------------------------------------------+
| sourceChain         : [32]byte             |                                        32 bytes |
+---------------------+----------------------+-------------------------------------------------+
| importedInputs      : []TransferableInput  |                  4 + size(importedInputs) bytes |
+---------------------+----------------------+-------------------------------------------------+
| outs                : []EVMOutput          |                            4 + size(outs) bytes |
+----------+----------+----------------------+-------------------------------------------------+
                                             |    80 + size(importedInputs) + size(outs) bytes |
                                             +-------------------------------------------------+
```

## Credenciales

Las credenciales solo pueden ser de un tipo: `SECP256K1Credential`. Cada credencial está emparejada con una entrada. El orden de las credenciales coincide con el orden de las entradas.

### SECP256K1 Credential

Una credencial [secp256k1](https://github.com/ava-labs/avalanche-docs/tree/94d2e4aeddbf91f89b830f9b44b4aa60089ac755/build/cryptographic-primitives/README.md#cryptography-in-the-avalanche-virtual-machine) contiene una lista de firmas recuperables de 65 bytes.

#### Contenido de una credencial SECP256K1

* **`TypeID`** es la identificación para este tipo. Es `0x00000009`.
* **`Signatures`** es un conjunto de firmas recuperables de 65 bytes. El orden de las firmas debe coincidir con los índices de firmas de la entrada.

#### Especificación Gantt de una Credencial SECP256K1

```text
+------------------------------+---------------------------------+
| type_id         : int        |                         4 bytes |
+-----------------+------------+---------------------------------+
| signatures      : [][65]byte |  4 + 65 * len(signatures) bytes |
+-----------------+------------+---------------------------------+
                               |  8 + 65 * len(signatures) bytes |
                               +---------------------------------+
```

#### Especificación Proto de una credencial SECP256K1

```text
message SECP256K1Credential {
    uint32 typeID = 1;             // 4 bytes
    repeated bytes signatures = 2; // 4 bytes + 65 bytes * len(signatures)
}
```

## Transacción Firmada

Una transacción firmada contiene un `AtomicTx` no firmado y credenciales.

### Que Contiene una Transacción Firmada

Una transacción firmada contiene un `CodecID`, `AtomicTx` y `Credentials`.

* **`CodecID`** La única identificación actual válida del codec es `00 00`.
* **`AtomicTx`** es una transacción atómica, como ya se explicó.
* **`Credentials`** es un conjunto de credenciales. Cada credencial corresponde a la entrada en el mismo índice en el AtomicTx

### Especificación Gantt de una Transacción Firmada

```text
+---------------------+--------------+------------------------------------------------+
| codec_id            : uint16       |                                        2 bytes |
+---------------------+--------------+------------------------------------------------+
| atomic_tx           : AtomicTx     |                          size(atomic_tx) bytes |
+---------------------+--------------+------------------------------------------------+
| credentials         : []Credential |                    4 + size(credentials) bytes |
+---------------------+--------------+------------------------------------------------+
                                     |   6 + size(atomic_tx) + len(credentials) bytes |
                                     +------------------------------------------------+
```

### Especificación Proto de una Transacción Firmada

```text
message Tx {
    uint16 codec_id = 1;                 // 2 bytes
    AtomicTx atomic_tx = 2;              // size(atomic_tx)
    repeated Credential credentials = 3; // 4 bytes + size(credentials)
}
```

## UTXO

Una UTXO o "Unspent Transaction Output" es una representación independiente de la salida de una transacción.

### Que Contiene Una UTXO

Un UTXO contiene un `CodecID`, `TxID`, `UTXOIndex`, `AssetID` y `Output`.

* **`CodecID`** El único `CodecID` válido es `00 00`
* **`TxID`** es la identificación de una transacción de 32 bytes. Los ID de transacción se calculan tomando sha256 de los bytes de la transacción firmada.
* **`UTXOIndex`** es un int que especifica la salida en la transacción especificada por **`TxID`** por la que fue creado este UTXO.
* **`AssetID`** es una matriz de 32 bytes que define a qué activo hace referencia este UTXO.
* **`Output`** es el objeto de salida que creó este UTXO. La serialización de las Salidas fue definida anteriormente.

### Especificación Gantt de una UTXO

```text
+--------------+----------+-------------------------+
| codec_id     : uint16   |                 2 bytes |
+--------------+----------+-------------------------+
| tx_id        : [32]byte |                32 bytes |
+--------------+----------+-------------------------+
| output_index : int      |                 4 bytes |
+--------------+----------+-------------------------+
| asset_id     : [32]byte |                32 bytes |
+--------------+----------+-------------------------+
| output       : Output   |      size(output) bytes |
+--------------+----------+-------------------------+
                          | 70 + size(output) bytes |
                          +-------------------------+
```

### Especificación Proto de una UTXO

```text
message Utxo {
    uint16 codec_id = 1;     // 02 bytes
    bytes tx_id = 2;         // 32 bytes
    uint32 output_index = 3; // 04 bytes
    bytes asset_id = 4;      // 32 bytes
    Output output = 5;       // size(output)
}
```
